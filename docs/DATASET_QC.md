# Process Lab

The workbench Process Lab page is available at:

```text
http://127.0.0.1:8765/#process
```

It provides lightweight checks for the current video and candidate set:

- an interactive frame viewer with play, previous/next, frame slider, and
  synchronized stage tiles
- a run selector tied to Architecture Lab manifests
- the selected run's ordered pipeline with each stage's expected QC outputs
- frame size and frame count
- candidate ROI count
- discovery suggestion count
- ROI area and equivalent diameter in pixels
- physical diameter when `pixel_size_microns` is present in the dataset manifest
- trace noise summary
- event-count distribution
- framewise max brightness and saturation summaries when available
- simple rigid drift summaries when available
- discovery artifact-cue burden
- generated missed-neuron proposal and ROI artifact-risk tables when attached
- evidence-map thumbnails

Use this page when a run looks suspicious in Review. It is meant to connect a
visual failure mode to the exact pipeline stage that produced, or should have
produced, the relevant intermediate output.

## Recommended Workflow

1. Select the same run that you are inspecting in Review or Architecture Lab.
2. Use the synchronized frame slider to inspect raw frames and generated stage
   outputs side-by-side.
3. Toggle missing outputs when you need to know which pipeline stages are still
   only planned metadata.
4. Check ROI-size, trace-noise, event-density, saturation, and drift warnings
   before interpreting parameter changes.
5. Use the Discovery and Artifact Triage tables to decide whether to loosen
   thresholds, suppress artifacts, or add missed-neuron review targets.

The page is designed to catch practical issues before parameter tuning:

- ROIs that are much smaller than expected soma size
- too many missed-neuron suggestions relative to detected ROIs
- many artifact-like discovery suggestions
- noisy traces or sparse event support
- drift large enough to make weak candidates suspicious
- saturation/bright-blob burden that may explain false positives

Review data generated by `tools/generate_neuron_review_app.groovy` includes a
`qc` block with frame-mean brightness, frame-max brightness, saturation, and
rigid drift estimates. These checks are conservative diagnostics; they do not
modify the raw video or hide candidates.

## Pipeline-Linked QC

Process Lab now uses the selected Architecture Lab run as its context. The
pipeline panel shows the sequential stages, their artifact flow, and the QC
outputs each stage declares in `neurobench.pipeline_catalog`. Outputs already
present in the current review data are highlighted; missing outputs remain
visible so it is clear what a future run should export for stronger diagnosis.

The main QC viewer is a synchronized stage grid. The raw video tile is always
shown first. Generated stage outputs appear beside it when the run manifest
declares browser-readable `artifacts.intermediates[]` entries with
`frame_pattern` or static `file` paths. Stages without generated intermediates
remain visible as placeholders, and the "missing outputs only" toggle makes it
easy to see what still needs to be exported.

The browser now accepts app-relative paths, generated-run paths, absolute
project paths under `Outputs/NeuronReview/<dataset>/app`, and cross-dataset app
paths. Broken media tiles are replaced with explicit missing-artifact messages
instead of silently rendering blank images.

## Discovery And Artifact Triage

Process Lab can now read a generated `analysis/proposal_analysis.json` artifact
from the selected Architecture Lab run. This file contains two explicit tables:

- missed-neuron proposals ranked by discovery score, event support, local
  coherence, compactness, repeated activity, and artifact penalties
- artifact-risk ROIs ranked by artifact score, background correlation, weak
  local coherence, footprint size, elongation, border proximity, and labels

Generate and attach those artifacts with:

```bash
python3 tools/build_proposal_analysis.py \
  --review-data Outputs/NeuronReview/calcium_rest_cropped/app/review_data.json \
  --annotations Outputs/NeuronReview/calcium_rest_cropped/app/annotations.json \
  --architecture-runs Outputs/NeuronReview/calcium_rest_cropped/app/architecture_runs.json \
  --run-id current_review_pipeline
```

The command also writes spreadsheet-friendly TSV files under `analysis/`.
If the JSON artifact is not present, the dashboard falls back to lightweight
browser-side estimates from the embedded review data.

To attach a TIFF stack as an intermediate stage output:

```bash
python3 tools/export_intermediate_frames.py \
  --input-tif Outputs/HighPass/calcium_rest_cropped/calcium_rest_cropped_hp_gaussian_sigma06f_float32.tif \
  --out-dir Outputs/NeuronReview/calcium_rest_cropped/app/intermediates/current_review_pipeline/temporal_highpass_gaussian \
  --architecture-runs Outputs/NeuronReview/calcium_rest_cropped/app/architecture_runs.json \
  --run-id current_review_pipeline \
  --stage-id temporal_highpass_gaussian \
  --label "Temporal high-pass"
```

This makes the page useful as both a dataset audit and a pipeline sanity check:
if a warning points to drift, nonuniform background, saturation, impulse noise,
or missed-neuron burden, the user can immediately see which stage in the
current architecture is supposed to handle that failure mode.
